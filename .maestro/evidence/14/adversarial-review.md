# Adversarial review of #14 — convert Maestro workflows into reusable workflows

## Findings

1. **bug**: `maestro-implement.yml` job-gate silently skips satellite callers whose triggering event isn't `issues`/`issue_comment`/`pull_request`.
   - Location: `.github/workflows/maestro-implement.yml:51-54`
   - What's wrong: The `if:` expression has three OR branches, each keyed to `github.event_name == 'issues'|'issue_comment'|'pull_request'`. In a `workflow_call`'d run, `github.event_name` is the **caller's** event. A satellite that wires its shim to, say, `workflow_dispatch` (the obvious "run Maestro now" button), `repository_dispatch`, `schedule`, or anything else evaluates to false on all three branches and the job is skipped with no log line and no failure. The added comment ("trust gate below still applies — same author_association checks fire") is half-true: the checks fire when the satellite's event happens to be one of those three, and silently no-op the workflow otherwise.
   - User-observable consequence: A satellite owner pins Maestro at `v0.1.0`, writes a `workflow_dispatch` shim that calls `manasgarg/maestro/.github/workflows/maestro-implement.yml@v0.1.0` with `maestro_ref: v0.1.0`, clicks Run, sees "All checks have passed" with zero work done, and has no diagnostic.
   - Suggested fix: Either add an explicit `inputs.maestro_ref != ''` OR-branch (with its own author_association check on whatever the caller's event provides), or document loudly that satellite shims must trigger from one of the three supported event types. Needs Implementer judgement on the trust model for non-issue caller events.

2. **bug**: `maestro-review.yml` job-gate silently skips satellite callers whose triggering event isn't `pull_request`.
   - Location: `.github/workflows/maestro-review.yml:34-35`
   - What's wrong: The `if:` reads `github.event.pull_request.author_association`. When called from a satellite via any non-PR caller event (e.g., a satellite that wraps the reviewer in a `workflow_dispatch` to manually re-review an already-merged PR), `github.event.pull_request` is null, the `contains(...)` returns false, the job is skipped silently. Same shape as finding 1 but harder to trip because PR review naturally fires on PR events; still a real footgun.
   - User-observable consequence: Same as finding 1 — a satellite shim that calls the reviewer from any non-PR event no-ops with a green check.
   - Suggested fix: Either restrict the documented caller surface to `pull_request` only and assert that early, or add an `inputs.maestro_ref != ''` OR-branch with a satellite-appropriate gate.

3. **risk**: Prompts loaded from `$PROMPTS_DIR/prompts/*.md` still contain hardcoded relative paths to other Maestro spec files; in satellite mode those paths resolve into the calling repo's tree, not into `.maestro-src/`.
   - Location: prompts referenced by `.github/workflows/maestro-implement.yml:100`, `.github/workflows/maestro-review.yml:69`, `.github/workflows/maestro-learn.yml:119`
   - What's wrong: `prompts/implementer.md:5` ("You may always read `DESIGN.md`"), `prompts/implementer.md:104` ("pass it the contents of `prompts/adversarial-reviewer.md` plus the path to the captured patch"), and `prompts/synthesizer.md:85,95` (`python3 tools/build_learnings_index.py`) are all relative-path references. In satellite mode the agent runs with cwd = satellite repo, so `DESIGN.md`, `prompts/adversarial-reviewer.md`, and `tools/build_learnings_index.py` do not exist at those paths — they live under `.maestro-src/`. The only signal the wrapper injects is the one-line hint `- maestro spec checkout: \`$PROMPTS_DIR/\``. The agent is expected to remap every internal reference from that hint alone.
   - User-observable consequence: First satellite run fails or degrades quietly — the Implementer can't spawn the Adversarial Reviewer (or spawns it without the prompt), the Synthesizer can't regenerate `INDEX.md`, the agents hit "file not found" for `DESIGN.md` and continue without the principles context. The tests in this PR are structural greps and won't catch any of it.
   - Suggested fix: Either rewrite the three prompts to accept a `$MAESTRO_SRC` env/placeholder and templatize them at compose time, or have the wrapper prepend a more forceful binding ("All Maestro spec paths — DESIGN.md, prompts/, tools/ — live under `$PROMPTS_DIR/`; never read them from the working tree"). The current single bullet is too easy for the model to miss.
