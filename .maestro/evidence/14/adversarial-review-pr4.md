# Adversarial review of #14 — PR 4: upstream learning routing

## Findings

1. **bug**: `git push` in `tools/upstream_learning.sh` has no credential helper set up, so the scheduled-satellite path fails.
   - Location: `tools/upstream_learning.sh:66-108` (clone via plain `https://github.com/manasgarg/maestro.git`, then `git push -u origin "$BRANCH"`)
   - What's wrong: The clone URL is `https://github.com/manasgarg/maestro.git` with no token embedded. In CI, `GH_TOKEN` env makes `gh auth status` and `gh api` work, but plain `git push` over HTTPS does not consult `GH_TOKEN`. There is no `gh auth setup-git`, no `url.insteadOf` credential injection, and no token rewrite of the remote URL. PR 3's `maestro-rollout.yml` explicitly does this (`git config --global url."https://x-access-token:${GH_TOKEN}@github.com/".insteadOf "https://github.com/"`); this PR omits it. In an interactive session a prior `gh auth login` would have installed the credential helper, but the scheduled satellite path the README and the new `MAESTRO_UPSTREAM_PAT` secret were explicitly added for has neither.
   - User-observable consequence: When a scheduled `maestro-learn.yml` run in a satellite produces a workflow-level learning with `MAESTRO_UPSTREAM_PAT` set, the script clones fine, commits fine, then `git push` errors with "could not read Username for 'https://github.com'" and the helper exits before opening the PR. The whole second path the PR ships is broken in CI.
   - Suggested fix: Either embed the token in the clone URL (`https://x-access-token:${GH_TOKEN}@github.com/manasgarg/maestro.git` when `GH_TOKEN` is set), or run `gh auth setup-git` before `git push`, or add a `git config url.…insteadOf` mirroring PR 3.

2. **bug**: `tests/test_upstream_learning_flow.sh` section 8's `awk` range scans well past the secret and matches any later `required: false`, so it doesn't actually check that `MAESTRO_UPSTREAM_PAT` is the one marked optional.
   - Location: `tests/test_upstream_learning_flow.sh:75` — `awk '/MAESTRO_UPSTREAM_PAT:/,/^[^[:space:]]/' "$WF" | grep -q 'required: false'`
   - What's wrong: The awk address-range end pattern `/^[^[:space:]]/` matches the first line starting with a non-whitespace character — but in `maestro-learn.yml` the `MAESTRO_UPSTREAM_PAT:` declaration is on line 45 (indented under `secrets:`), and within its block `required: false` does appear, so today this happens to pass. The problem: `awk` evaluates `/pat1/,/pat2/` as inclusive of any line that matches either pattern, and the first match for `MAESTRO_UPSTREAM_PAT:` is *also* matched by patterns later in the file (the comment at line 182 and the env line at line 189, both indented but containing the literal token name) — so the awk range actually re-opens for those lines and continues scanning past them, which means a stray `required: false` on a wholly unrelated input later in the file would satisfy the grep even if the secret were flipped to `required: true`. The workflow already contains 5 `required: false` lines elsewhere. Verified by counting.
   - User-observable consequence: A regression that makes `MAESTRO_UPSTREAM_PAT` required would not be caught by the test — Maestro's own scheduled `maestro-learn.yml` runs (which don't supply the secret) would silently start failing in production while CI stays green.
   - Suggested fix: Parse the YAML properly, or restrict to lines indented deeper than the secret key, or grep the two-line window directly (e.g., `grep -A2 'MAESTRO_UPSTREAM_PAT:' | grep -q 'required: false'`).

3. **risk**: The synthesizer/intake prompts tell agents to `curl … | bash` the helper from the pinned ref, but the URL hardcodes `manasgarg/maestro` while the satellite may have been installed from a fork.
   - Location: `prompts/synthesizer.md:25` and `prompts/intake.md:61` — `curl -fsSL https://raw.githubusercontent.com/manasgarg/maestro/$(cat .maestro/version)/tools/upstream_learning.sh`
   - What's wrong: `.maestro/version` is a pinned ref (tag/sha), but the *org/repo* is hardcoded. If a satellite was installed from a fork (e.g., a private vendored copy at `myorg/maestro`), the version ref won't exist at `manasgarg/maestro` and the curl returns 404 (with `-f`) or the wrong content. The script itself then also targets `manasgarg/maestro` for the clone and PR — so the upstream route is hard-bound to the canonical name regardless of where the satellite came from. There is no override and no detection.
   - User-observable consequence: A satellite installed from a Maestro fork can't route upstream at all — the curl 404s or the PR opens against the wrong repo. The PR's own "Background you should rely on" notes mention "the satellite is checked out from a fork rather than the canonical name" as a failure mode worth checking; the diff doesn't address it.
   - Suggested fix: Read the upstream org/repo from a config (e.g., `.maestro/upstream` or an env var `MAESTRO_UPSTREAM_REPO`) and substitute it into both the curl URL and the script's clone/PR target; default to `manasgarg/maestro`.

4. **risk**: The existing-file collision check in `upstream_learning.sh` races against in-flight PRs from other satellites.
   - Location: `tools/upstream_learning.sh:82-89`
   - What's wrong: The `[ -f "$DEST" ]` test runs against a shallow clone of `main`. Two satellites that independently produce a learning with the same slug, both run within the same window, will both pass the check (since neither's PR is merged yet), both push `upstream/<slug>-<ts>` branches, and both open PRs. The second PR to merge will conflict on `INDEX.md` and silently overwrite the first satellite's file (same filename) — defeating the "refuses to overwrite" guarantee the script and test #4 advertise. The synthesizer's "check existing learnings via `gh api …/contents/.maestro/learnings`" precheck has the same race window: it only sees merged files.
   - User-observable consequence: Two satellites racing on the same slug silently clobber one another's learnings at merge time; the human reviewer of the second PR has no signal that there's a competing PR.
   - Suggested fix: Also list open PRs with `gh pr list --search "Upstream learning: $SLUG"` and abort if any are open; or include enough provenance (`source:` from the original satellite) in the filename so independent extraction produces different slugs.

5. **smell**: Test #5's `grep -q 'gh api user'` is a substring check that wouldn't catch the relevant regression.
   - Location: `tests/test_upstream_learning_flow.sh:44`
   - What's wrong: The grep matches if the literal string `gh api user` appears anywhere — including in a comment, or in an unrelated invocation. The real claim of the test ("commit author would not match PR author" if missing) requires the script to (a) read the gh user identity AND (b) actually pass it to `git config user.name`/`user.email`. The current script does both, but only the first is structurally tested; removing the `git config user.email …+…@users.noreply.github.com` line would leave the test green and silently regress the PR-3 fix the comment cites.
   - User-observable consequence: A future refactor could remove the `git config user.email` line and the test would still pass; commits would then come from a fallback identity (or no identity, depending on global git config).
   - Suggested fix: Add a second grep for the `git config user.email` line with the `…+…@users.noreply.github.com` pattern (or assert both `gh api user --jq .login` and `gh api user --jq .id` appear).
